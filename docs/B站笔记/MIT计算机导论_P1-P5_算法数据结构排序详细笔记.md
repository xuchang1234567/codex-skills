# MIT 计算机导论 P1-P5：算法、数据结构与排序

## 阶段目标

前五讲建立算法课的基本思维：先明确问题和接口，再选择数据结构，最后分析正确性与资源成本。不要把算法学习理解成背代码；同一个问题换一种数据结构，运行时间和实现复杂度可能完全不同。

## P1：Algorithms and Computation

### 核心问题

算法是解决一类问题的有限步骤描述。学习一个算法时至少要回答四件事：输入是什么，输出是什么，为什么一定得到正确结果，随着输入规模增长需要多少时间和空间。

### 正确性与效率

“程序能运行”不等于“算法正确”。应先写出不变量或归纳依据，再考虑优化。复杂度分析通常关注输入规模趋于大时的增长趋势，忽略常数和低阶项，但工程上仍需关注常数、内存、缓存和实现语言。

### 学习动作

对一个查找问题分别写线性扫描和更快的方案，记录它们对输入是否有序、是否需要额外存储等前提。每次比较都写出：正确性依据、最好/平均/最坏情况和空间需求。

## P2：Data Structures and Dynamic Arrays

### 数据结构的视角

数据结构不是某种固定代码，而是数据和操作的组织方式。先列出需要支持的操作，再选择结构：访问、查找、插入、删除、遍历和扩容的优先级不同，最优选择也不同。

### 动态数组

动态数组用连续空间保存元素，支持按索引快速访问；空间不足时申请更大的区域并搬移旧元素。单次扩容可能是 O(n)，但若容量按倍数增长，连续追加的摊销成本可以达到 O(1)。

这里要区分：

- 单次操作成本：某一次扩容可能很慢；
- 摊销成本：在一长串追加操作上平均分摊扩容开销；
- 空间成本：预留容量换取更少的扩容次数。

### 实践题

实现动态数组时测试容量为 0、连续追加、删除末尾、插入开头和大量扩容。记录元素搬移次数，观察为什么插入开头通常比末尾昂贵。

## P3：Hashing

### 基本机制

哈希表通过哈希函数把键映射到数组位置，目标是在平均意义下快速查找、插入和删除。冲突不可完全避免，需要开放寻址或链式结构等处理方法。

### 必须理解的条件

- 哈希函数应尽量把键均匀分布；
- 负载因子过高会使冲突增加；
- 扩容需要重新分布已有键；
- 平均 O(1) 不代表任何输入下都 O(1)；
- 删除操作在开放寻址中不能简单清空，否则可能破坏后续查找路径。

### 设计判断

如果任务需要按键快速查找，哈希表通常合适；如果需要有序遍历、范围查询或稳定顺序，哈希表未必是最佳结构。应把操作需求和数据特征放在一起判断。

## P4：Problem Session 2

这一讲用于把前面内容变成解题能力。推荐采用以下流程：

1. 写清楚输入、输出和约束。
2. 先给出能保证正确的朴素方案。
3. 找出重复工作或昂贵操作。
4. 选择动态数组、哈希或其他结构消除重复。
5. 证明关键不变量。
6. 计算时间和空间复杂度。

做题时不要一上来猜数据结构。先问“我要反复查询什么”“元素是否需要保持顺序”“是否允许额外空间”“输入是否动态变化”。这些问题往往比代码模板更能决定方案。

## P5：Linear Sorting

### 为什么可能快于比较排序

比较排序通过元素之间的大小比较确定顺序，通常受到比较模型下的下界约束。若键具有特殊结构，例如整数范围有限、位数固定或可以按数字位处理，就可以利用计数、桶或基数等信息减少比较。

### 适用前提

所谓线性排序并非无条件 O(n)。必须说明：

- 键的取值范围是否有限；
- 范围是否与 n 同数量级；
- 是否需要额外数组；
- 是否要求稳定排序；
- 多位键的位数是否固定。

### 复杂度判断

分析线性排序时，要把遍历输入、遍历键范围、桶数量、位数和辅助空间分别写出来。若键范围远大于元素数量，计数结构可能反而浪费空间；若键是字符串或多字段对象，还要说明每次处理的成本。
